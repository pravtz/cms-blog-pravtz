## Codebase Patterns
- `useToast()` returns `{ toast, dismiss }` — call `toast({ variant: 'success'|'error'|'warning'|'info', title: '...' })` (not `addToast`, not `type`).
- `Modal` prop is `open` (not `isOpen`). Other props: `onClose`, `title`, `footer` (ReactNode).
- Badge variants: `'default' | 'primary' | 'success' | 'warning' | 'error' | 'info'` — no `neutral` variant.
- Test files (`src/__tests__/`) are excluded from `tsconfig.json` — they use top-level `await import()` which requires `target: ES2022` not set in the Next.js tsconfig. Vitest handles ESM natively so type-checking tests separately is fine.
- Integration tests mock `@/lib/db` with `vi.mock()` returning an in-memory SQLite DB (created via `createTestDb()` in `helpers/db.ts`). Never call `getDb()` directly in test setup.
- SQLite `datetime('now', ...)` returns `'YYYY-MM-DD HH:MM:SS'` (no T, no timezone). JavaScript's `new Date()` ISO format is `'YYYY-MM-DDTHH:MM:SS.sssZ'`. String comparison between these formats is unreliable (space < 'T' in ASCII). Always insert test timestamps using `new Date(...).toISOString()` to match what the route's `windowStart` comparison expects.
- Vitest coverage: exclude `db.ts`, `mdx.ts`, `rate-limit.ts` from coverage (require external deps); target files are `auth.ts`, `rbac.ts`, `utils.ts`, `auth-middleware.ts`, `v1-helpers.ts`.
- The German `ß` does not decompose in NFD normalization — it stays as `ß` and gets replaced by `-` in `slugify()`. Expected: `'Straße'` → `'stra-e'`.

- Blog `PostGrid.tsx` is a `'use client'` component; receives `initialPosts` + `total` from SSG page; LoadMore fetches from blog's own `/api/posts` route (which proxies to admin's `/api/blog/posts`).
- Blog's `next.config.js` uses `images.remotePatterns` with `{ protocol: 'https'/'http', hostname: '**' }` to allow any remote cover image.
- Blog home page uses `export const revalidate = 60` (ISR) for automatic page refresh.
- Blog feed page at `/blog` uses `searchParams` (Next.js server component) for SSR + ISR; interactive filtering via `router.push` re-renders the server component. Client components using `useSearchParams()` must be wrapped in `<Suspense>`.
- Blog filter-based pages (category/tag/year) stay indexed; only `?q=` search query pages get `noindex` to prevent duplicate content issues.
- `lib/mdx.ts` exports `sanitizeMDX(content)` (sync, server-side sanitization before DB store) and `renderMDX(content)` (async, unified pipeline → Shiki → HTML, for blog rendering).
- Shiki v4 is installed. Server-side: `import { codeToHtml } from 'shiki'`. Client-side (browser): `import('shiki/bundle/web')` via dynamic import inside useEffect.
- `hast-util-to-html` is available as a transitive dep (from react-markdown). Use `processor.parse()` + `await processor.run()` to get the hast tree, then `toHtml(hast)` — do NOT use `processor.process()` without rehype-stringify.
- Blog post page: admin `/api/blog/posts/[slug]` renders MDX server-side and returns `content_html`; blog uses `dangerouslySetInnerHTML` — no shiki needed in blog.
- CSS Module `:global()` is needed to style `dangerouslySetInnerHTML` HTML content (e.g. `.content :global(h2) { ... }`).
- `NEXT_PUBLIC_ADMIN_URL` env var is for client-side fetches to admin; `ADMIN_URL` (no PUBLIC prefix) is server-only.
- Unified pipeline packages available as transitive deps (via react-markdown): unified, remark-parse, remark-gfm, remark-rehype, hast-util-to-html. Import them with dynamic `await import()` in server utilities.
- sanitize-html is installed in admin. Import as `import sanitizeHtml from 'sanitize-html'`.

- v1 API: all routes in `apps/admin/src/app/api/v1/` use `applyRateLimit()` + `handleOptions()` from `lib/v1-helpers.ts`. Rate limiter is in `lib/rate-limit.ts` (Redis sliding window, fail-open). Redis client is in `lib/redis.ts` (lazy init, fail-open on error).
- v1 API security headers are set in `buildV1Headers()` in `lib/v1-helpers.ts` — CSP, nosniff, DENY, HSTS, Cache-Control, CORS, X-RateLimit-*.
- OpenAPI spec is served at `/api/v1/docs?format=json`; Swagger UI HTML at `/api/v1/docs`.

- Monorepo: npm workspaces + Turborepo. Run workspace commands with `npm run <cmd> --workspace=@nexus/<app>`.
- DB layer lives in `apps/admin/src/lib/db.ts` — `getDb()` returns a singleton better-sqlite3 instance with WAL mode.
- Migrations are run in `runMigrations()` inside `db.ts`; always use `IF NOT EXISTS` / `ON CONFLICT DO UPDATE` patterns.
- All Next.js API routes that use better-sqlite3/bcryptjs need `export const runtime = 'nodejs'` and `export const dynamic = 'force-dynamic'`.
- Auth helpers (hashPassword, verifyPassword, generateToken, generateRefreshToken) live in `apps/admin/src/lib/auth.ts`.
- CSS Module files sit next to their component (e.g. `setup.module.css` next to `page.tsx`).
- Design tokens are CSS custom properties on `:root` in `apps/admin/src/app/globals.css`; full Onyx theme implemented in US-03.

- `Modal` and `Toast` are `'use client'` components; Modal implements focus-trap + Escape handler + body-scroll-lock.
- `ToastProvider` must wrap the app (or a subtree) to use `useToast()` hook. Add it to `layout.tsx` when needed.
- Auth middleware for API routes: `requireAuth(request)` / `requireRole(request, ...roles)` in `apps/admin/src/lib/auth-middleware.ts` — returns `{ payload }` on success or a `NextResponse` on failure.

- Component `style` prop only available if explicitly in the interface — Card, Badge, etc. don't accept arbitrary HTML attrs; use a wrapper `<div>` in stories instead.
- Editor components live in `apps/admin/src/components/Editor/`; exported from barrel `Editor/index.ts`. Use `@uiw/react-codemirror` + `@codemirror/lang-markdown` + `@codemirror/theme-one-dark`.

- Toolbar format operations use `view.dispatch({ changes, selection })` directly — pass viewRef to toolbar component.
- `react-markdown` custom components: use `ExtraProps` from `react-markdown` + `ClassAttributes`/`HTMLAttributes` from React to type component props without `any`.
- `@typescript-eslint/no-explicit-any` eslint disable comments will cause lint errors if the rule isn't in project ESLint config — use proper types instead.
- posts/categories/tags API routes: slugify uses NFD normalization + regex for accented chars; always add `IF NOT EXISTS` indexes.
